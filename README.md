# Добавлялка треков в плейлист Я.Музыки

Владелец логинится в свою Я.Музыку прямо в браузере и создаёт ссылки вида
«сбор треков на вечеринку» — каждая ведёт в свой плейлист. Дальше ссылкой
можно поделиться с кем угодно: получатели просто вставляют ссылку на трек
или альбом в форму, без какой-либо регистрации — все треки уходят в плейлист
от имени владельца.

Работает через неофициальную библиотеку [`yandex-music`](https://github.com/MarshalX/yandex-music-api)
(официального открытого API для правки плейлистов у Яндекса нет).

## Как это устроено

- `core.py` — вся логика: разбор ссылок, добавление треков/альбомов,
  список плейлистов аккаунта.
- `store.py` — маленькое JSON-хранилище (`state.json`): токен, полученный
  через OAuth, и созданные шаринг-ссылки (slug → плейлист).
- `web.py` + `templates/*.html` — веб-оболочка на Flask:
  - `/admin` — вход по паролю, логин в Я.Музыку, создание/удаление ссылок;
  - `/s/<slug>` — публичная форма добавления треков для конкретной ссылки.
- `get_token.py` — получить токен вручную (нужно только для legacy-бутстрапа,
  см. ниже).

Токен один на всё приложение (аккаунт того, кто прошёл OAuth в `/admin`).
Гости, переходящие по `/s/<slug>`, никак не авторизуются — они просто
добавляют треки в плейлист, привязанный к этой ссылке.

## Установка

```bash
pip install -r requirements.txt
cp .env.example .env
```

В `.env` обязательно нужны:

```
ADMIN_PASSWORD=придумай-пароль
FLASK_SECRET_KEY=  # python -c "import secrets; print(secrets.token_hex(32))"
```

Без `ADMIN_PASSWORD` раздел `/admin` отключён (503). Без стабильного
`FLASK_SECRET_KEY` сессия админки слетает при каждом перезапуске процесса —
для локальной проверки это не страшно, для прод — обязательно задать.

## Запуск

```bash
python web.py
# открой http://127.0.0.1:5000 — страница с описанием сервиса,
# внизу ссылка «Владелец плейлиста? Войти в админку»
```

1. Залогинься паролем из `ADMIN_PASSWORD`.
2. Нажми «Войти через Яндекс» — покажется ссылка и код (OAuth Device Flow).
   Открой ссылку на любом устройстве, введи код, подтверди вход — админка
   сама обнаружит подтверждение (поллинг) и обновится.
3. Выбери плейлист из своего аккаунта, впиши название (например, «Вечеринка
   у Макса») и нажми «Создать ссылку». Ссылку `/s/<slug>` можно скопировать
   и отправить кому угодно.
4. Плейлистов и ссылок может быть сколько угодно — под каждый повод/плейлист
   удобно завести отдельную ссылку. Ненужные — удаляются кнопкой в таблице.

### Публичная форма (`/s/<slug>`)

Вставляешь ссылку на трек → трек в плейлисте. Дедупликация есть: если трек
уже в плейлисте, повторно не добавит.

Можно вставить и ссылку на альбом целиком (без `/track/…`, например
`https://music.yandex.ru/album/3192570`) — перед добавлением покажется
диалог со списком всех треков альбома (уже добавленные помечены «уже есть»),
и только после подтверждения все новые треки уйдут в плейлист.

Под формой показывается текущее содержимое плейлиста: название (кликабельное,
ведёт на плейлист в Я.Музыке) и список треков — обновляется после каждого
добавления. Есть поиск по списку (фильтр по названию и исполнителю, на
клиенте, без лишних запросов).

## Legacy-бутстрап без браузерного логина

Если не хочется проходить OAuth в `/admin` (например, headless-деплой без
удобного доступа к браузеру), токен можно получить один раз вручную:

```bash
python get_token.py
```

и вписать в `.env`:

```
YM_TOKEN=ПОЛУЧЕННЫЙ_ТОКЕН
```

Это только подставляет токен по умолчанию — плейлист всё равно назначается
через ссылку, созданную в `/admin` (`YM_PLAYLIST` в `.env` сейчас ни на что
не влияет в веб-режиме, это исторический дефолт). Без хотя бы одной ссылки
из `/admin` публичной формы не будет.

## Прод — Docker

`python web.py` поднимает dev-сервер Flask (только для локальной проверки).
В прод — через Docker: там gunicorn на `0.0.0.0:8000`.

Разово:

```bash
cp .env.example .env      # вписать ADMIN_PASSWORD и FLASK_SECRET_KEY
docker compose up -d --build
```

`docker-compose.yml` монтирует `./data` как volume под `state.json` — без
этого токен и ссылки терялись бы при каждой пересборке образа.

Приложение на `http://<сервер>:8000`. Проверка живости — `GET /healthz`.
Логи: `docker compose logs -f`. Остановить: `docker compose down`.

Без compose, если удобнее чистый docker:

```bash
docker build -t ym-adder .
docker run -d --name ym-adder -p 8000:8000 \
  -v "$(pwd)/data:/app/data" \
  -e ADMIN_PASSWORD="<пароль>" \
  -e FLASK_SECRET_KEY="<секрет>" \
  ym-adder
```

Важное про секреты и прод:

- Токен, пароль админки и секрет сессий передаются только через переменные
  окружения / `.env`, в образ они не зашиваются. Файл `.env` не коммить
  (он уже в `.gitignore`), как и `data/`/`state.json`.
- Наружу лучше выставлять не голый 8000, а через nginx/Caddy с HTTPS —
  особенно важно для `/admin`, т.к. сессия админки держится на cookie.
- Плейлисты общие в рамках своей ссылки: всё, что вводят гости по
  конкретному `/s/<slug>`, падает в привязанный к ней плейлист. Сама ссылка
  не защищена паролем — доступ регулируется тем, кому она разослана.

## Безопасность админки

- Пароль (`ADMIN_PASSWORD`) сравнивается через `hmac.compare_digest`
  (устойчиво к timing-атакам), сама `/admin/login` ограничена — 10 неудачных
  попыток с одного IP за 5 минут, дальше блокирует и верный пароль тоже.
  Ограничение живёт в памяти процесса, не переживает перезапуск и не общее
  между воркерами gunicorn — заметно поднимает порог для брутфорса, но не
  строгая гарантия; для чего-то серьёзнее нужен внешний rate-limit (nginx).
- Сессионная cookie — `HttpOnly` и `SameSite=Lax` всегда; `Secure` включается
  флагом `SESSION_COOKIE_SECURE=1` в `.env`, когда сервис стоит за HTTPS
  (иначе cookie в браузере не выставится вовсе на голом http — оставь
  выключенным для локальной проверки).
- `state.json` (содержит боевой токен) пишется с правами `0600` — на Linux
  (в т.ч. в Docker) это ограничивает чтение владельцем процесса; на Windows
  такого понятия нет, так что при локальной разработке файл читаем как
  обычный.
- Ни один шаблон не отключает автоэкранирование Jinja — названия плейлистов,
  лейблы ссылок и т.п. безопасно показываются даже если содержат `< >` и т.д.

## Оговорки

- API неофициальный: Яндекс может поменять его в любой момент, тогда библиотеку
  придётся обновить (`pip install -U yandex-music`).
- Не гоняй тысячи запросов подряд — можно словить ограничение аккаунта.
